# Prompt 25: Content - Documentation for Monetization Features

## Objective
Create clear, comprehensive documentation for all new monetization features to help users, creators, and developers understand and use them effectively.

## Explanation
Good documentation is a product feature in itself. For monetization, it's especially critical. Creators need to understand how to price their work and how they get paid. Developers need to understand how your API billing works. Clear documentation reduces confusion, minimizes support tickets, and encourages adoption of your paid features.

## Instructions
1.  **Structure the Documentation:**
    *   In your existing `/docs` directory or on your documentation site, create a new top-level section called "Monetization".
    *   Under this section, plan out the individual articles you will need.

2.  **Write Content for Creators:**
    *   **Article: "Selling Your Skills"**
        *   Explain how to set a price for a skill.
        *   Discuss the fee structure (what percentage the platform takes).
        *   Explain the payout process, schedule, and requirements (e.g., tax forms).
    *   **Article: "Referral Program Guide"**
        *   Explain how to find the referral code.
        *   Detail the commission rate and the terms (e.g., commission on first purchase only).
        *   Provide tips on how to effectively share the code.

3.  **Write Content for Developers (API Consumers):**
    *   **Article: "API Subscription Plans"**
        *   Detail the different subscription tiers.
        *   Clearly list the usage limits (e.g., API calls per month) and features for each plan.
        *   Explain what happens when a user goes over their limit.
    *   **Article: "Understanding API Billing"**
        *   Explain how API calls are counted.
        *   Describe the monthly billing cycle.
        *   Provide instructions on how to update payment methods via the billing portal.
    *   **Article: "API Authentication"**
        *   Reiterate how to generate and use API keys securely.

4.  **Review and Publish:**
    *   Have someone unfamiliar with the features read the documentation to check for clarity.
    *   Ensure all links are correct.
    *   Publish the documentation and link to it from relevant places in your application UI (e.g., a "Read the Docs" link on the API keys page).
